# CPU SoC固件质量分析 - 架构设计说明书

## 1. Overview

**背景**：CPU SoC在产品与解决方案时，需要高度的软硬件协同能力，对业务场景满足度和协议遵从度的提出较高的要求，需要在黑盒测试方法外，构建更强的质量风险评估和测试设计能力的工程方法。

**定位**：
1. 不是一个泛泛的代码分析工具，也不是再造一个轮子，市面上有coverity, clang-tidy等工具在这方向已经做得很好。
2. 以业务导向的测试方法，建立“场景级”标准来指导测试方向，进行断言定义，市面的上固件分析工具大多没有这样的视角。

3、工程实践原则：架构先行，场景驱动，工程聚焦。

因此，要设计一个框架：对固件实施**多维度、多阶段**的深度分析，以**函数调用链为骨架**，中断、寄存器、状态机为构件，建立固件的**结构化知识库**以及模型导向的分析方法。其核心目的是：

- **质量风险评估**：通过调用链分析、中断并发分析、寄存器访问审计、状态机完整性检查等方法，识别黑盒测试难以覆盖的边界场景和隐蔽缺陷，量化场景满足度缺口
- **测试设计指导**：基于固件实际行为模型，自动推导测试关注点——哪些中断可能冲突、哪些寄存器组合可能非法、哪些状态转换路径未被覆盖，为白盒/灰盒测试提供输入
- **协议遵从度验证**：通过符号执行和可达路径分析，验证固件在特定输入下的行为是否符合协议规范

**典型场景与问题**：
1. **DDR 行为的协议一致性**——DDR 初始化/训练过程涉及数十个寄存器的严格时序配置和校准状态机的迭代收敛。寄存器配置错误或状态转换遗漏可能导致 DDR 无法稳定工作。传统黑盒测试只能看到最终的读写成败，无法定位到"哪个寄存器在哪个状态中被错误配置"这一根因。
   - **测试参考依据**：JEDEC DDR 协议规范（时序参数 tRCD/tCL/tRP 等）、SoC DDR 控制器数据手册（寄存器地址与字段定义）、校准状态机文档（状态转换表与判定条件）。
2. **外设协议交互的边界合规性**——SPI/I2C 等外设通信依赖中断驱动的数据收发、寄存器组的正确配置（速率、模式、极性）以及主循环状态机对协议进度的管理。非法组合（如在中断中误写速率寄存器）可能导致协议时序错乱，黑盒测试难以枚举这类组合异常。
   - **测试参考依据**：SPI/I2C 总线协议规范（时序波形、帧格式）、SoC 外设寄存器手册（控制/状态/数据寄存器位域定义）、外设驱动状态机构造文档。
3. **中断并发下的资源共享安全性**——主循环与多个中断服务函数共享全局变量时，C 语言层面无保护机制。关键变量在中断上下文被修改后，主循环读取到的值可能违反时序预期（如 DMA 缓冲区在传输中间被 ISR 写入），这类竞态条件在黑盒测试中极难复现。
   - **测试参考依据**：中断向量表与优先级配置（NVIC 寄存器设置）、共享变量清单（变量名→读写函数→中断级映射）、临界区保护规范（关中断/信号量使用约束）。

## 2. 问题与架构决策

| # | 问题 | 决策 |
|---|------|------|
| 2.1 | 嵌入式固件质量分析的核心抓手是什么？通用静态检查工具（Coverity、clang-tidy）擅长检测内存泄漏、空指针等通用缺陷，但固件的特殊问题（中断并发、寄存器误操作、状态机逻辑跳转）需要不同的分析视角 | **以函数调用链为骨架**：函数调用关系是固件逻辑最自然的组织方式。在此基础上融合多个嵌入式特有要素——中断（并发模型）、寄存器（硬件接口）、状态机（逻辑语义）——挂载到调用链上，形成**统一的程序依赖图**。这一视图既是分析的中间表示，也是最终可查询的结果 |
| 2.2 | 与 Coverity、clang-tidy 等工具复用同一分析结果有何不同？ | Coverity/clang-tidy 是**单遍扫描式**的通用缺陷检查器，每次运行独立分析源码，输出零散的告警；本项目的定位是**多阶段、增量式、以调用图为中心的固件专属分析平台**——每阶段产出结构化的中间结果（调用图、寄存器表、事件模型等），下游阶段可在其上叠加分析（如安全审计、事件流追踪、符号执行），最终形成可检索、可追溯的固件知识库 |
| 2.3 | 为什么要以 "场景" 而非 "缺陷" 作为分析视角？ | **场景驱动**——固件的质量不取决于没有缺陷，而取决于在**关键业务场景下的行为是否符合预期**（如在特定中断序列下协议握手是否成功、在寄存器组合异常时状态机是否安全）。场景级分析需要先建立固件的行为模型，再在其上定义断言和测试关注点。这与传统工具 "扫描→告警" 的模式有本质区别 |
| 2.4 | 固件分析涉及预处理、AST 解析、二进制反汇编、安全建模、符号执行等多个领域，耦合在一起难以维护和扩展 | 采用 **6 阶段流水线**架构，每阶段职责单一、输入输出明确，通过 `data_pipeline_config.json` 控制启停，新增阶段继承 `BaseStage` 并实现 `run()` |
| 2.5 | 函数调用关系不是树结构，同一调用点可能有多个目标，需保留调用地址等多维信息 | 使用 `networkx.MultiDiGraph`（多重有向图），同一对节点允许多条边（以 `key` 区分），序列化格式统一为 `{nodes: [{id, attributes}], edges: [{source, target, key, attributes}]}` |
| 2.6 | 单一数据源不可靠——AST 可能丢失间接调用，二进制可能丢失符号名 | **同时维护静态调用图和二进制调用图**，最终合并为统一图（`call_graph_unified.json`），加入寄存器节点，形成函数+数据依赖视图 |
| 2.7 | ARM Thumb 模式下函数边界检测困难，`BLX reg` 等间接跳转无法纯静态解析 | **优先使用 angr CFGFast**，通过符号执行和数据流分析解析间接跳转；不可用时降级到 capstone + 符号表 |
| 2.8 | 分析结果需要对外暴露，支持 SPA 和第三方集成 | 使用 **FastAPI** 构建 REST API，`fastapi-offline` 保证离线 Swagger 文档可用，统一前缀 `/api/v1/sw_model/`，按 `static/binary/modeling/sym_execution/risks` 划分 namespace |
| 2.9 | 开发调试需同时启动前端静态服务器和后端 API 服务器，部署繁琐 | **共享端口 5007**——`run_server.py` 直接启动 FastAPI 并在 `/` 挂载前端静态文件，消除跨域和端口管理 |
| 2.10 | 拓扑图中节点可能需运行时切换普通节点/容器角色，GoJS 默认 Node/Group 不兼容 | **统一输出 `isGroup: true`**——后端转换器将所有节点标记为 Group，前端 `GoJSCommon` 使用统一 Group 模板渲染，拖拽直接设 `group` 属性即可嵌套 |

## 3. 流水线设计

### 3.1 设计思路

质量分析不是一次扫描就能完成的——不同粒度的分析需要不同的输入、不同的工具、不同的计算开销。如果所有分析耦合在一个阶段中，会带来三个问题：

1. **工具链脆弱**：某个工具出错（如 angr 安装失败）导致整个分析不可用
2. **数据不可复用**：每次分析都重新解析源码，无法增量构建
3. **分析目标混乱**：安全检查和代码结构提取耦合在同一个 pass 中，无法独立演进

**流水线的设计原则**：前序阶段为后续阶段准备结构化数据，每个阶段聚焦于**一个分析目标**，输出**可独立验证的中间产物**。这使得下游阶段可以在不同时间点、用不同工具、针对不同目标叠加分析。

### 3.2 数据流设计

```
compile_commands.json ──→ Stage 1 ──→ 归一化源码（展开宏、保留行号）
                                  │
                                  ↓
                            Stage 2 ──→ 源码级结构化知识（函数、调用链、寄存器、状态机等）
                                  │
                                  ↓
                            Stage 3 ──→ 二进制级结构化知识（符号、反汇编、DWARF）
                                  │       → 统一图（合并 Stage 2 + Stage 3 + 寄存器）
                                  ↓
                            Stage 4 ──→ 场景模型 + 安全审计 + 协议一致性
                                  │
                                  ↓
                            Stage 5 ──→ 符号执行验证（协议一致性复现）
                                  │
                                  ↓
                            Stage 6 ──→ 动态测试用例设计（下游系统输入）
```

各阶段的输入输出总览：

| 阶段 | 职责（质量分析视角） | 输入 | 核心输出 |
|------|---------------------|------|----------|
| Stage 1 — 源码预处理 | 展开宏、包含头文件，保留行号 | `compile_commands.json` | `.i` / `.ii` 预处理文件 |
| Stage 2 — AST 分析 | 提取函数、变量、类型、中断、寄存器等高层结构 | 预处理文件 | `functions.json`, `call_graph.json`, `registers.json` 等 |
| Stage 3 — ELF 分析 | 二进制级别符号、反汇编、DWARF 提取 | ELF 文件 | `binary_functions.json`, `call_graph_binary.json` |
| Stage 4 — 场景模型 + 协议一致性 | 场景建模 + 断言审计 + 协议规范比对 | Stage 2 + 3 结果 + spec_model | `event_architecture.json`, `security_report.json`, `protocol_conformance.json` |
| Stage 5 — 符号执行 | angr 动态路径探索与约束求解，复现/验证协议一致性发现 | ELF 文件 + Stage 4 报告 | `symbolic_report.json`, `traces/` |
| Stage 6 — 动态测试用例设计 | 将 Stage 4/5 发现转化为动态测试用例，供下游系统执行 | Stage 4 + 5 报告 | `test_spec.json`, `coverage_gaps.json` |

关键设计决策：

- **Stage 3 并行于 Stage 1→2**：二进制分析不依赖预处理和 AST，可以独立运行，也意味着即使源码不可用仍可从 ELF 直接分析
- **统一图在 Stage 3 中构建**：因为 Stage 3 同时接触到二进制视角和静态视角，是合并两者的自然汇合点
- **Stage 4 叠加在 Stage 2+3 之上**：事件建模同时需要源码级语义和二进制级地址映射

### 3.3 阶段依赖关系

```
Stage1 ──→ Stage2 ──┐
                      ├──→ Stage4 ──→ Stage5 ──→ Stage6
Stage3 ──────────────┘
```

### 3.4 场景驱动视角：流水线产出作为场景模型的数据基础

从场景分析的视角看，流水线的各阶段不是在孤立地分析代码，而是在**逐层构建场景模型**：

| 阶段 | 对场景建模的贡献 | 场景分析示例 |
|------|-----------------|-------------|
| Stage 1→2 | 建立源码级要素清单（函数、ISR、寄存器、状态机） | 识别出 `SPI_Transfer()` 访问了哪些寄存器，在哪些源文件中 |
| Stage 3 | 补充二进制级验证（调用关系是否被优化改变） | 确认 `SPI_Transfer()` 实际调用了 `HAL_SPI_Transmit()` (即使被 inline) |
| Stage 4 | 将要素关联为场景模型——中断序列 + 调用路径 + 状态转换；比对协议规范产出风险发现 | 构建场景："PendSV 中断中调用 SPI_Transfer() 时，状态机处于 STATE_BUSY 是否安全？"；MR6 寄存器值是否超出协议范围 |
| Stage 5 | 验证场景的可达性和断言，复现协议一致性发现 | 符号执行验证该场景是否真可达，推导输入条件生成测试用例 |
| Stage 6 | 将已验证的风险转化为具体测试规格，供下游动态测试执行 | 为 DDR5 VREF 越界发现生成测试序列：写入 MR6→读取确认→边界值遍历 |

所谓 "场景"，在固件语境下定义为一个三元组：**{触发事件, 调用路径, 状态上下文}**。流水线的各阶段依次为这三个维度提供数据支撑。

### 3.5 前置条件

固件工程需提供两样东西：
- **ELF 文件**（firmware.elf）—— 通过标准构建流程（make / cmake）生成
- **compile_commands.json**—— CMake 项目自动生成，非 CMake 项目可用 `bear` 或 `compiledb` 工具补全

### 3.6 配置方式

流水线的启停、参数、模块映射由 `data_pipeline_config.json` 统一配置，允许按需开关每个阶段。配置文件的格式本身也是架构的体现——每个阶段是一个独立配置块，与流水线设计原则保持一致。示例配置参见 [data_pipeline_config.json](./data_pipeline_config.json)。

## 4. Stage 1 — 源码预处理 (Preprocessor)

### 4.1 要解决的问题

C 语言源码中有两样东西阻碍直接进行结构化分析：

1. **宏**：`#define` 将大量语义信息隐藏在符号替换中（如寄存器地址、状态枚举值），AST 分析看到的是宏名而非展开后的具体值
2. **头文件包含**：函数定义可能分散在数十个头文件中，单文件分析无法建立完整的调用视图

预处理就是用编译器自身的展开能力，将宏展开、头文件内联，生成一份**归一化后的中间源码**，让后续 AST 分析看到的是完整的、可解析的 C 语法结构。

### 4.2 为什么保留行号

预处理通常输出不包含原始源位置的文件（`-P` 标志），但这对质量分析是致命的——分析到的问题如果不关联到原始源码位置，就无法指导测试和修复。因此预处理阶段强制保留 `#line` 行号标记，使 AST 分析能够追溯到函数定义的原始文件和行号。

## 5. Stage 2 — AST 分析 (AST Analysis)

### 5.1 要解决的问题

归一化后的源代码包含了固件逻辑的全部信息，但编译器只关心能否生成目标码，**不会告诉我们**：
- 哪些函数调用了哪些函数？
- 哪些全局变量被哪些函数读写？
- 固件注册了哪些中断服务函数？
- 哪些内存地址被当作寄存器操作？
- 哪些 enum + switch 构成了状态机？

AST 分析的使命正是从源码的结构化表示中**提取这些语义信息**，将其从源码文本转化为**可查询的结构化数据**。

### 5.2 提取项与质量分析意图

| 提取项 | 分析意图 | 对质量评估的价值 |
|--------|----------|-----------------|
| **函数与调用链** | 固件逻辑骨架——谁调用了谁 | 识别不可达函数（死代码）、中间层缺失（架构断层）、调用深度异常（栈风险） |
| **全局变量** | 函数间共享状态 | 识别跨中断共享的全局变量，评估并发风险 |
| **类型定义** | struct/union/enum 布局 | 验证内存对齐、字段偏移是否与硬件规格一致 |
| **中断服务函数（ISR）** | 异常入口点 | 识别未注册的中断、优先级配置冲突 |
| **寄存器访问** | `*(volatile uintXX_t *)ADDR` 模式 | 清单化固件与硬件的交互点，评估寄存器配置是否正确 |
| **状态机** | enum + switch 模式 | 验证状态转换图是否完整，是否存在死状态或不可达状态 |
| **函数调用链（Call Graph）** | 上述所有项的串联骨架 | 作为统一图的基础结构，将中断、寄存器、状态机挂载到调用链上 |

### 5.3 引擎选择策略

首选 **libclang**（通过 clang Python 绑定），因为它是和编译器同一级别的 C 解析器，能正确处理复杂宏展开、属性标注、内联汇编。备用 **tree-sitter**，适用于 libclang 安装不可用的环境，但 tree-sitter 不做语义分析，无法区分变量声明和函数调用。

### 5.4 对场景建模的价值

Stage 2 的产出是场景模型的 **要素层**——它回答了 "固件中有什么"：哪些函数、哪些中断服务函数、哪些寄存器、哪些状态机。这些要素在后续 Stage 4 中被组合成完整的场景模型（{触发事件, 调用路径, 状态上下文}），因此 Stage 2 的提取项越多、越准确，场景模型的覆盖度就越高。

## 6. Stage 3 — ELF 分析 (ELF Binary Analysis)

### 6.1 要解决的问题

AST 分析有两个固有盲区：

1. **编译器优化**：内联函数、尾调用优化、链接时优化会改变实际的函数调用关系，AST 看到的是源码级的调用，而非实际执行时的调用
2. **间接调用**：通过函数指针的调用在 AST 层面无法确定目标函数（只有变量名，没有具体值）

ELF 分析从已编译的二进制出发，看到的是**编译器最终生成的指令序列**，不受宏和优化的干扰。同时通过反汇编识别 `BL` / `BLX` 指令，捕获源码级不可见的间接调用目标。

### 6.2 调用图构建策略

| 场景 | 策略 | 适用条件 |
|------|------|----------|
| angr 可用 | angr CFGFast — 通过符号执行和数据流分析解析函数边界和间接跳转 | 大多数开发/分析环境 |
| angr 不可用 | capstone 反汇编 + 符号表识别 BL/BLX 指令，仅识别直接调用 | 受限环境或快速分析 |

### 6.3 统一图（Unified Graph）的设计意图

将静态调用图（Stage 2）和寄存器列表合并为一个 `MultiDiGraph`，是因为：

- **函数与寄存器共同构成了固件与硬件交互的完整画面**——函数代表逻辑，寄存器代表物理接口
- **统一图是后续分析的基础中间表示**——Stage 4 的事件建模、Stage 5 的符号执行、前端可视化都在此图上叠加分析
- 函数节点标记 `node_type="function"`，寄存器节点标记 `node_type="register"`，两类节点共存于同一图中，边表示调用关系或数据依赖

### 6.4 对场景建模的价值

Stage 3 为场景模型提供了 **验证层**——Stage 2 提取的调用路径可能被编译器优化改变（内联、尾调用），Stage 3 的二进制调用图验证了 "实际执行的调用路径是什么"。统一图将两者合并，确保场景模型中的调用路径维度同时包含源码意图和二进制事实。

## 7. Stage 4 — 场景模型构建与安全审计 (Scenario Modeling)

### 7.1 要解决的问题

Stage 2 和 Stage 3 产出了大量的结构化信息（函数、调用链、寄存器、状态机），但它们是**离散的**——没有回答固件层面的核心问题：**哪些场景是关键的？每个场景的行为是否符合预期？**

嵌入式固件的核心运行模型是**中断驱动**——主循环 + 多个中断服务函数交错执行。一个 "场景" 就是：**在什么中断上下文中、经过哪些函数调用路径、在什么状态下、访问了哪些寄存器**。

离散要素不建模就不能回答这类问题。同时，中断驱动的并发模型带来了独特的质量风险：

1. **中断并发**：一个全局变量可能被主循环和中断同时访问，但 C 语言层面没有任何并发保护机制
2. **优先级反转**：高优先级中断等待低优先级中断使用的资源
3. **状态机安全**：中断可能在状态机处于中间状态时触发，导致非法状态转换

**测试目标**：验证离散的结构化信息是否能组合为完整的场景模型，并基于模型断言判断固件行为是否符合设计预期。只有建立场景模型，才能从 "扫描到 N 个告警" 提升到 "场景 S 在条件 C 下有风险 R" 的可追溯判断。

**测试参考依据**：

| 参考来源 | 为断言提供的依据 | 对应场景维度 |
|----------|-----------------|-------------|
| 中断向量表 + NVIC 优先级配置 | 每个中断的优先级层级、嵌套关系、触发入口 | 触发事件 |
| 外设驱动状态机文档 | 状态枚举值含义、合法转换边、触发条件 | 状态上下文 |
| SoC 寄存器手册（地址→位域） | 每个寄存器的读写属性、合法取值集合、复位值 | 资源依赖 |
| 统一图（Stage 3 产出） | 函数间调用的实际路径（验证过的，而非 AST 推导的） | 调用路径 |
| 临界区保护设计规范 | 哪些变量需要保护、保护手段（关中断/信号量） | 断言判定准则 |

### 7.2 场景模型构建

| 维度 | 方法 | 输出（场景要素） |
|------|------|-----------------|
| 中断映射 | 将 ISR 符号与中断向量表地址关联，确定每个中断的触发路径 | 场景的 **触发事件** 维度 |
| 事件注册表 | 识别回调注册模式（如 `register_callback`、`NVIC_SetPriority`） | 场景的 **调用路径** 入口 |
| 状态机模型 | 将 Stage 2 识别的状态机与对应的事件处理函数关联 | 场景的 **状态上下文** 维度 |
| 数据流图 | 在统一图上追踪全局变量的读写函数集合 | 场景的 **资源依赖** 维度 |

场景模型的三元组 **{触发事件, 调用路径, 状态上下文}** 在每个维度上由具体的 Stage 4 输出数据填充。

### 7.3 基于场景的断言与审计

场景模型建立之后，可在其上定义**断言**——即场景的预期行为规范：

| 断言类型 | 面向的问题 | 对测试设计的指导 |
|---------|-----------|----------------|
| 中断并发断言 | "ISR_A 和主循环同时访问变量 X 时是否有保护？" | 指导设计临界区测试用例 |
| 状态机合法性断言 | "中断在 STATE_MID 触发时状态转换是否合法？" | 指导设计非法状态转换测试 |
| 寄存器配置断言 | "场景执行后寄存器 R 的值是否符合协议要求？" | 指导设计寄存器采样验证点 |
| 栈深度断言 | "最深层嵌套的中断路径栈使用是否超过阈值？" | 指导栈配置验证 |

### 7.4 输出与下游服务

- `event_architecture.json`：场景模型，包含中断映射、调用路径、状态上下文
- `security_report.json`：安全审计报告，包含每个断言的检查结果

Stage 4 完成的是**静态风险识别**。其中部分断言（如寄存器配置合法取值、状态机转换边完整性）可以直接通过静态分析完成测试；但另一部分断言（如中断路径可达性、场景触发条件）依赖具体运行时输入，静态分析只能标记为 "可疑"，无法给出确定性结论。

因此 Stage 4 的结构化输出不仅是一份报告，更是 Stage 5 符号执行和动态测试用例设计的**输入规格**——它告诉下游："哪些断言值得验证、验证什么路径、需要什么约束条件"。这种 "静态识别 → 动态验证" 的分工是流水线设计的核心原则之一。

## 8. Stage 5 — 动态符号执行 (Dynamic Symbolic Execution)

### 8.1 要解决的问题

前四个阶段都是**静态分析**——不执行固件，仅从源码和二进制中推导信息。静态分析的固有缺陷是**误报**：分析出的危险路径可能在运行时因输入条件不满足而**实际不可达**。

符号执行通过将固件二进制加载到符号执行引擎（angr）中，用符号值代替具体输入，探索固件在**所有可能输入下的执行路径**。相比静态分析，它能回答：
- 这条 "可疑路径" 是否真正可达？（验证/排除误报）
- 到达这条路径需要什么输入条件？（生成测试用例）
- 执行到目标点时，哪些内存位置的值是可预测的？（约束求解）

**测试目标**：将 Stage 4 场景模型中的断言从 "假设性" 转化为 "可验证的"，为每条断言提供可达性结论和输入约束——即断言是否成立、成立需要什么条件。

**符号执行的适用性**：并非 Stage 4 的所有断言都需要符号执行。断言的验证手段与其性质相关：

| 断言性质 | 示例 | 验证手段 | 适用性说明 |
|---------|------|---------|-----------|
| 静态可判定 | "寄存器 R 的第 3 位是否在中断中被写入了非法值？" | Stage 4 静态审计直接完成 | 依赖已知取值集合，无需运行时输入 |
| 路径可达性 | "中断 A 到目标函数 B 的调用路径是否成立？" | **符号执行** | 依赖运行时控制流，静态分析只能假设 |
| 输入依赖 | "场景 {中断A, 路径B, 状态C} 是否可在实际输入下触发？" | **符号执行** | 需要约束求解，推导具体输入条件 |
| 时序依赖 | "中断 A 和主循环同时访问变量 X 是否真的发生？" | **动态测试**（符号执行可辅助） | 依赖时钟级时序，符号执行可缩小探索范围 |

Stage 5 专注于上述第二、三类断言——即**符号执行最能发挥作用、且 Stage 4 无法独立给出结论**的问题。这种针对性确保了符号执行的计算资源投入到最高价值的验证目标上。

**测试参考依据**：

| 参考来源 | 为验证提供的依据 | 验证内容 |
|----------|-----------------|---------|
| Stage 4 产出的安全审计报告 | 告警列表中的每条断言及其关联的路径（ISR 入口 → 目标函数） | 验证路径可达性，排除不可达告警 |
| 场景模型的三元组 {事件, 路径, 状态} | 场景的精确约束：什么中断触发、经过哪些函数、状态机处于什么值 | 推导满足场景的输入约束 → 生成测试用例 |
| 协议规范（I2C/SPI/DDR 等时序参数） | 协议层面的合法取值区间、时序边界 | 验证固件在边界值处的行为是否符合协议预期 |
| ELF 二进制文件 | 固件的实际指令序列、内存布局、符号地址 | 符号执行的加载目标和路径探索范围 |

### 8.2 对场景建模的价值

场景模型（Stage 4 产出）中的断言是**假设性的**——"如果中断 A 在状态 S 触发，变量 X 可能被并发访问"。但这条路径是否真的可达？真正触发需要什么输入条件？静态分析无法回答。

符号执行将场景模型的断言转化为可验证的问题：

| 场景验证场景 | 符号执行如何回答 | 对测试设计的意义 |
|-------------|-----------------|-----------------|
| Stage 4 告警验证 | "路径 X 从 ISR 入口到目标函数是否可达？" | 排除不可达告警 → 降低误报率 |
| 场景可达性 | "达到场景 {中断A, 路径B, 状态C} 需要的输入约束是什么？" | 直接生成白盒测试用例 |
| 协议合规性检查 | "用符号值模拟协议输入，固件在规格边界处的行为是否符合预期？" | 验证场景断言是否成立 |
| 缓冲区溢出 | "场景中 memcpy(dst, src, len) 的 len 是否可能超过 dst 边界？" | 确认场景安全边界 |

### 8.3 路径爆炸的应对

固件通常跑在资源受限的 MCU 上，代码规模较大，全路径符号执行会面临路径爆炸问题。应对策略（按优先级）：

1. **目标导向**：只从 Stage 4 告警地址出发进行后向切片，缩小探索范围
2. **Veritesting 模式**：angr 的混合执行模式，在动态执行和符号执行间切换
3. **探索深度限制**：设置最大路径长度，超出则标记为 "未验证"

## 9. Stage 6 — 风险审计与聚合 (Risk Audit & Aggregation)

### 9.1 要解决的问题

前五个阶段完成了从静态分析到符号执行的逐层精化，产出了固件的结构化风险清单（Stage 4 协议一致性发现 + 安全审计）和路径可达性验证结果（Stage 5 符号执行复现）。但这些发现分散在多个报告中，且缺乏对"已验证风险"和"遗留风险"的明确划分。

Stage 6 不做测试用例生成——测试用例设计是下游系统的职责。Stage 6 的职责是：

1. **聚合记录**：将 Stage 4（静态识别）和 Stage 5（符号执行验证）的发现按协议/版本归并，如实记录每个发现的风险等级、来源阶段、验证状态
2. **风险分类**：标记每个发现属于"已验证风险"（Stage 5 复现确认）还是"遗留风险"（仅静态识别，未在符号执行中复现或无法复现）
3. **如实输出**：下游系统消费这份风险注册表，自行决定如何设计动态测试用例

**核心原则**：Stage 6 是"镜子"而非"设计师"——只记录不创造，确保下游看到的是经过多阶段分析后最完整、最可信的风险全景。

### 9.2 数据输入

| 输入来源 | 数据内容 | 用途 |
|---------|---------|------|
| Stage 4 `protocol_conformance.json` | 协议一致性发现（状态机、时序、寄存器、错误处理、跨寄存器依赖） | 提供初始风险清单 |
| Stage 4 `security_report.json` | 全部静态断言发现 | 补充安全/质量类风险 |
| Stage 5 `symbolic_report.json` | 入口点执行状态、InspectTrigger 记录、风险分布统计 | 标记哪些发现已在符号执行中复现 |
| Stage 5 `traces/` | 逐入口点符号执行详情 | 为每个已复现风险提供可追溯的上下文（symbolic vars、寄存器快照、路径约束） |

### 9.3 输出模型

#### 9.3.1 风险注册表（risk_registry）

**文件路径**：`data/risks/{protocol}_{version}/protocol_conformance/risk_registry.json`

```jsonc
{
  "protocol": {
    "name": "string",                    // 协议名称，如 "DDR5"
    "version": "string"                  // 协议版本，如 "JESD79-5C"
  },
  "generated_at": "string",              // ISO 8601 时间戳
  "summary": {
    "total_risks": 0,                    // 风险总数
    "verified": 0,                       // 已验证（Stage 5 复现确认）
    "residual": 0,                       // 遗留（仅 Stage 4 识别）
    "severity_distribution": {
      "critical": 0, "high": 0, "medium": 0, "low": 0, "info": 0
    }
  },
  "risks": [
    {
      "risk_id": "string",               // 风险唯一标识
      "source": "stage4" | "stage5",     // 首次发现的阶段
      "finding_ref": "string",           // 引用原始报告中的 finding ID
      "risk_type": "state_machine_completeness"
                 | "timing_constraint_audit"
                 | "register_config_audit"
                 | "error_handling_coverage"
                 | "cross_register_dependency",
      "severity": "critical" | "high" | "medium" | "low" | "info",

      "spec_ref": "string",              // 协议规范引用
      "location": {                      // 风险位置
        "file": "string",
        "function": "string"
      },
      "verification": {                  // 验证详情（仅 verified 时有此字段）

          "r1": "string",
          "pc": "string"
        }
      }
    }
  ]
}
```

#### 9.3.2 遗留风险清单（residual_risks）

**文件路径**：`data/risks/{protocol}_{version}/protocol_conformance/residual_risks.json`

从 `risk_registry` 中筛选出 `status = "residual"` 的条目，独立输出以便下游系统直接关注未验证风险。

```jsonc
{
  "protocol": {
    "name": "string",
    "version": "string"
  },
  "total_residual": 0,
  "note": "These risks were identified by Stage 4 static analysis but NOT reproduced by Stage 5 symbolic execution. They require downstream dynamic testing for further validation.",
  "risks": [
    {
      "risk_id": "string",
      "finding_ref": "string",
      "risk_type": "string",
      "severity": "string",
      "message": "string",
      "spec_ref": "string",
      "location": {
        "file": "string",
        "function": "string"

                        | "symbolic_timeout"         // 符号执行超时

                        | "static_only"               // 仅静态可判定，不适用符号执行
    }
  ]
}
```

### 9.4 下游系统消费约定

Stage 6 的输出是下游动态测试系统的**输入素材**，而非可执行规格。下游系统按以下方式消费：

| 下游系统 | 消费方式 | 说明 |
|---------|---------|------|
| **动态测试设计工具** | 读取 `risk_registry.json`，按 `severity` 和 `status` 优先级排序，自行设计测试序列 | `verified` 风险有 trace 可复现上下文；`residual` 风险需额外探索 |
| **HIL 测试平台** | 读取 `residual_risks.json` 中 `register_config_audit` 类风险，转化为硬件引脚驱动序列 | 关注寄存器越界写入等可硬件验证的风险 |
| **人工测试分析** | `risk_registry.json` + `traces/` 提供完整证据链，测试人员据此编写测试用例 | trace 中的 symbolic vars 和 constraints 可直接指导输入设计 |
