# DDR5 初始化固件 — 功能说明书

## 1. 概述

本文档描述 `fw_samples/src/` 和 `fw_samples/lib/` 中 DDR5 初始化固件模块的功能规格、架构设计及其与本项目 `sw_model` 分析流水线的对应关系。

本模块是一个**简化的 DDR5 初始化控制器固件**，实现了 JEDEC JESD79-5C 标准中定义的 DDR5 初始化流程（§3.3 Initialization Procedure），作为 `sw_model` 流水线的**被测对象（DUT）**，用以实例化验证本项目 "以 spec_model 结构化协议知识作为质量分析参考源" 的方法论。

---

## 2. 与 spec_model 的对应关系

`spec_model` 项目对 DDR5 JEDEC 规范进行了结构化提取，建立了三层模型：元图 → 结构空间 → 行为空间。本固件模块的每个要素均可映射到 spec_model 中的对应实体：

| 固件要素 | 对应 spec_model 实体 | 质量分析用途 |
|---------|---------------------|-------------|
| `src/ddr/ddr5_regs.h` 中的 MR 定义 | object_entities → register | 验证固件配置的 MR 值是否在协议定义范围内 |
| `src/ddr/ddr5_cmds.h` 中的 CA 命令编码 | generalized_interfaces → CA_Bus | 验证固件发送的命令序列是否符合协议规定的顺序 |
| `src/ddr/ddr5_timing.h` 中的时序参数 | behavior_constraints → timing | 验证固件时序控制是否满足 tRCD/tCL/tRP 等最小值 |
| `src/ddr/ddr5_training.h` 中的状态机 | behavior_constraints → state_machines | 验证状态转换是否完整，是否存在死状态或跳转缺失 |
| `src/ddr/ddr5_init_controller.c` 中的初始化序列 | scenarios → initialization path | 验证固件实现的初始化路径与协议定义的规范路径是否一致 |

## 3. DDR5 初始化场景

### 3.1 场景定义

**场景名称**：DDR5 初始化和训练

**场景三元组**：

| 维度 | 内容 |
|------|------|
| 触发事件 | SoC 上电复位完成后，DDR 控制器开始初始化 |
| 调用路径 | `ddr5_initialize()` → 11 个状态函数 → 各训练例程 |
| 状态上下文 | `ddr5_init_state_t` 枚举（11 个状态 + 错误状态） |

### 3.2 状态机说明

固件实现的初始化状态机共 11 个状态，按 JEDEC JESD79-5C §3.3 规定的顺序执行：

```
RESET → CKE_HIGH → MR_CONFIG → VREF_TRAIN → ZQ_CALIB
  → DQ_TRAIN → CA_TRAIN → DCA_TRAIN → CRC_CONFIG
  → MR_VERIFY → READY
  (任何状态出错 → ERROR)
```

各状态对应的协议规范引用：

| 状态 | 协议规范引用 | 关键约束 |
|------|------------|---------|
| RESET | JESD79-5C §3.3.1 | RESET 低电平 ≥ 200μs，VDD/VDDQ 稳定 |
| CKE_HIGH | JESD79-5C §3.3.2 | CKE 高后等待 tXPR（≥ 512 tCK） |
| MR_CONFIG | JESD79-5C §3.4 | MR0–MR5 配置，MRW 间隔 ≥ tMRD（8 tCK） |
| VREF_TRAIN | JESD79-5C §3.3.4 + §3.4.7–3.4.8 | MR6–MR7 训练值，Vref 范围受 MR2 约束 |
| ZQ_CALIB | JESD79-5C §3.3.5 | ZQ_CL 后等待 tZQinit（≥ 1024 tCK） |
| DQ_TRAIN | JESD79-5C §3.4.9–3.4.10 + MPC | MR8–MR9 训练模式，写均衡 |
| CA_TRAIN | JESD79-5C §3.4.11 | MR10 CA 训练模式 |
| DCA_TRAIN | JESD79-5C §3.4.12 | MR11 占空比调整 |
| CRC_CONFIG | JESD79-5C §3.4.21–3.4.22 | MR21–MR22 CRC 模式 |
| MR_VERIFY | JESD79-5C §3.3.6 | MRR 回读验证所有已配置 MR |
| READY | — | 正常操作状态 |

### 3.3 已知的质量问题（用于验证分析能力）

固件代码中**有意引入**了几个典型的嵌入式固件质量问题，用以验证 `sw_model` 流水线的分析能力：

| # | 问题类型 | 位置 | 描述 |
|---|---------|------|------|
| 1 | 时序约束遗漏 | `ddr5_init_controller.c:_state_mr_verify()` | MR 重试时 `ddr5_write_reg(DDR_CTRL_MR_CMD, ...)` 后缺少 `DDR5_WAIT_CYCLES(DDR5_TMRD_CYCLES)`，违反了 JEDEC §5.2.17 的 tMRD 要求 |
| 2 | 状态机覆盖不完全 | `ddr5_state_names` 数组 | 数组第 11–14 个条目未定义（DDR5_STATE_ERROR=15 以后的条目缺失），可能导致状态名调试输出错误 |
| 3 | 寄存器配置顺序风险 | `_state_mr_config()` | MR3 在 MR5 之前配置，但 MR5 的 DFE_EN 依赖于 MR3 的 CRC 配置状态——硬件设计依赖未被代码注释覆盖 |
| 4 | 中断安全风险 | `_state_zq_calib()` | `DDR5_WAIT_CYCLES(DDR5_TZQINIT_CYCLES)` 为阻塞等待，若 ZQ 状态寄存器通过中断更新，轮询模式可能导致竞态 |
| 5 | 错误处理覆盖不足 | `_state_mr_verify()` 重试路径 | MR 回读失败后直接重新触发 MRW 写操作，但未检查 `DDR_CTRL_MR_CMD` 的 `MR_CMD_BUSY` 位——若上一次 MRW 还未完成，重试可能导致命令队列错误 |
| 6 | 边界值未校验 | `_state_vref_train()` | `ddr5_train_vref_dq()` 和 `ddr5_train_vref_ca()` 的返回值直接传递给 `DDR5_MR6_VREF_DQ_VAL()`，后者为 7-bit 掩码。若训练函数返回 128–255 之间的值，掩码会将非法值截断为合法值，导致训练错误被静默掩盖 |

## 4. sw_model 流水线分析映射

### 4.1 Stage 2 (AST 分析) — 从源码提取结构化信息

| 分析项 | DDR5 固件中的具体内容 | 对质量评估的意义 |
|--------|----------------------|-----------------|
| 函数 | `ddr5_initialize()`, `_state_mr_config()`, 等 14 个函数 | 构建调用链，识别训练例程的调用关系 |
| 全局变量 | `ddr5_state_names` 静态数组 | 识别状态名表是否完整（对应问题 #2） |
| 寄存器访问 | `ddr5_write_reg(DDR_CTRL_MR_CMD, ...)` 等 10+ 处 | 清单化 SoC 寄存器访问点，映射到协议定义的 MR 地址 |
| 状态机 | `ddr5_init_state_t` 枚举 + `ddr5_initialize()` 中的 switch-like 序列 | 验证状态枚举与代码路径的一致性 |
| 函数调用链 | `_state_mr_config()`→`_mrw_register()`, `_state_vref_train()`→`ddr5_train_vref_dq()` 等 | 构建场景的调用路径维度 |

### 4.2 Stage 3 (ELF 分析) — 二进制级别验证

| 分析项 | 验证目标 |
|--------|---------|
| 符号表 | 验证所有训练例程是否被编译器保留（或内联优化掉） |
| 函数边界 | 验证状态机的每个状态函数是否在预期的地址范围 |
| 二进制调用链 | 确认编译器优化后的实际调用路径（尤其是内联到 `ddr5_initialize()` 中的状态函数） |

### 4.3 Stage 4 (场景模型构建) — 断言生成与审计

| 断言类型 | 具体断言 | 参考依据 |
|---------|---------|---------|
| 状态机完整性 | "状态枚举是否有空条目导致 debug 输出错误？" | `ddr5_state_names` 数组 vs `ddr5_init_state_t` 枚举定义 |
| 时序约束 | "所有 MRW 后是否都等待了 tMRD？" | spec_model 中的时序约束 + DDR5_TMRD_CYCLES 使用点扫描 |
| 寄存器配置 | "MR0 到 MR22 的配置值是否在协议定义的合法范围内？" | spec_model register 实体中的合法取值集合 |
| 命令序列 | "初始化命令顺序是否符合 JEDEC §3.3 指定的路径？" | spec_model scenarios 中的规范路径 |
| 错误处理 | "错误状态 ERROR 是否有对应的恢复路径？" | 状态转换图的可达性分析 |

### 4.4 Stage 5 (符号执行) — 断言验证与测试用例生成

| 验证目标 | 符号执行方法 | 期望产出 |
|---------|-------------|---------|
| 问题 #1: tMRD 时序遗漏 | 符号化 `DDR_CTRL_TRAIN_STATUS`，探索 MR 重试路径 | 验证 `mrw_register→mrr_verify` 路径中 tMRD 等待是否被跳过 |
| 问题 #4: 中断安全 | 符号化中断标志位，探索 ZQ_CALIB 状态中的竞态 | 验证 `ddr5_read_reg(DDR_CTRL_ZQ_CTRL)` 的轮询是否与中断 ISR 冲突 |
| 问题 #6: Vref 边界截断 | 符号化 `ddr5_train_vref_dq()` 的返回值（0–255） | 验证当训练函数返回 >127 时，`DDR5_MR6_VREF_DQ_VAL()` 掩码是否静默截断错误值 |
| 场景可达性 | 符号化上电复位信号，探索 RESET→READY 完整路径 | 验证定义的状态序列是否全部可达 |

## 6. 工程实践原则

1. **架构先行**：DDR5 初始化固件按 JEDEC 规范的状态机结构组织代码，每个状态对应一个独立函数，与流水线 Stage 2 的 AST 分析能力对齐
2. **场景驱动**：固件的质量评估不是 "扫描代码找 bug"，而是 "验证 DDR5 初始化场景是否满足 JEDEC 协议规范"——这正是 Overview 中定位的核心区别
3. **工程聚焦**：代码中有意引入的 6 个质量问题，涵盖了嵌入式固件中最常见的风险类别（时序、状态机、寄存器、中断、边界值、错误处理），聚焦于质量和测试设计的实际工程痛点
